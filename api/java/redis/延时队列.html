# 异步消息队列

Redis 的list（列表）数据结构常用来作为异步消息队列使用，使用repush/lpush操作入队列，使用lpop和rpop来出队列。

> rpush notify-queue apple banana pear
(integer) 3
> llen notify-queue
(integer) 3
> lpop notify-queue
"apple"
> llen notify-queue
(integer) 2
> lpop notify-queue
"banana"
> llen notify-queue
(integer) 1
> lpop notify-queue
"pear"
> llen notify-queue
(integer) 0
> lpop notify-queue
(nil)

# 队列空了怎么办？

机制：客户端是通过队列的pop操作来获取消息，然后进行处理。处理完了再接着获取消息，在进行处理，如此循环往复，这便是作为队列消费者的生命周期。

问题：假设队列空了，客户端就会陷入pop的死循环，不停的pop，没有数据，接着在pop。这是浪费生命的空轮询。这就会导致客户端的cpu被拉高，redis的QPS也会被拉高。

方案：使用sleep来解决这种问题，让线程睡一会，睡个1s。

time.sleep(1) # python 睡 1s
Thread.sleep(1000) # java 睡 1s

# 队列延迟

问题：用上面的办法可以解决问题，但是会出现消息的延迟增大，，可以通过降低消息延迟的时间，但不是最好的方案；可以通过blpop/brpop,这两个b代表的是blocking，也就是阻塞读。

描述：阻塞读在队列里面没有数据的时候，会立即进入休眠状态，一旦数据到来，则立刻醒来。

# 空闲链接自动断开

问题：空闲连接问题；

如果线程一直阻塞在哪里，Redis的客户端连接就成了限制连接，闲置过久，服务器就会一般主动断开连接，减少闲置资源占用。这时候blpop和brpop会抛出异常来。所以需要捕获异常，还要重试。

# 锁冲突处理

问题：之前在讲到分布式锁的问题，但是没有提到客户端在处理请求时加锁没有成功。一般有三种策略来处理加锁失败。

  1. 直接抛出异常，通知用户稍后再重试
  2. sleep一会再重试
  3. 将请求转移至延迟队列，过一会再试
# 直接抛出特定类型的异常

这种方式比较适合由用户直接发起的请求，用户看到错误对话框后，会先阅读对话框的内容，再点击重试，这样就可以起到人工延迟

# 延时队列

延时队列可以通过Redis的zset（有序列表）来实现。我们将消息序列化成一个字符串作为zset的value，这个消息的到期处理时间作为score,让后用多线程轮询zset获取到期的任务进行处理，多个线程是为了保障可用性。万一挂了一个线程还有其他线程可以继续处理。因为有多个线程，所以需要考虑并发争抢任务，确保任务不能被多次执行。
